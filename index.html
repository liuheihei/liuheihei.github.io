TCP四次断开

发表于 2018-02-28 | 分类于 技术 |

请详细描述一下tcp四次断开的原理

关键词解释

全双工： 数据能在两个方向上同时传递
半关闭： TCP提供了一端结束数据发送之后还能接收另一端数据的能力。
MSL： (Maximum Segment Lifetime)报文最大生存时间。

四次关闭过程

一、主动关闭方生成ISN并向被动关闭方发送FIN、ISN，代表本地已经发送完所有数据，等待关闭。主动关闭状态由established变为FIN-WAIT1。
二、被动关闭方收到FIN、ISN后，回复ACK(ACK=ISN+1)，并等待本地数据发送完成，此时被动关闭状态由established变为CLOSE_WAIT，主动关闭状态由FIN-WAIT1变为FIN-WAIT2。
三、被动关闭方发送完所有数据后生成ISN并向主动关闭方发送FIN+ISN。此时被动关闭状态由CLOSE_WAIT变为LAST-ACK。
四、主动关闭方收到FIN+ISN后回复ACK(ACK=ISN+1)，此时状态为TIME-WAIT，等待2MSL的时间后关闭连接此时状态为CLOSED
五、被动关闭方收到ACK后关闭连接，状态变为CLOSED

连接断开

关于四次断开常见的问题

一、为什么TCP握手要三次、而断开要四次？
答： 由于TCP是全双工的方式，两端都需要等待本地数据发送完成后才能彻底断开连接，
二、为什么主动关闭方最后还要等待2MSL的时间而不直接关闭连接？
答：主动关闭方无法确认被动关闭方是否收到了最后的ACK，等待2MSL的原因是防止被动方最终的ACK没有收到，被动方反复发送FIN。TIME-WAIT状态的端口不能被使用

TCP三次握手

发表于 2018-02-27 | 分类于 技术 |

请详细描述一下tcp三次握手的原理

关键词解释

ISN(Initial Sequence Number)： 初始序列号32位计数器(0-4294967295)，系统在初次启动时会将该编号重置为0，每500ms增加64000，保证序列号不重复，(老式生成方法这种方式并不安全下面会介绍)。新式生成ISN方法，M代表生成的计数器编号，F代表函数(md5、md4)，IP四元祖(源目ip\port)。 ISN = M + F(SIP, SPort, DIP, DPort，)

MSS： 最大报文长度，在三次握手时双方为协商这个字段，以MSS较少的一方来决定。

三次握手的过程

一、请求端生成一个ISN序列号，并向接收连接端发送SYN、ISN、MSS大小。
二、接收端收到请求后，生成一个ISN序列号，回复ACK、SYN、ISN、MSS，其中ACK=请求端的ISN+1。
三、请求端收到接收端的请求后，回应ACK=接收端ISN+1

链接建立。

抓包时各个字段的含义

标志 3字符缩写 描述
S SYN 同步序号
F FIN 发送方完成数据传送
R RST 复位连接
P PSH 尽可能的将数据送往接收进程
. 以上四个标志位都置为0

问题

一、为什么使用计数器的方式生成ISN不安全？
假设有三个角色，服务器A、服务器B、攻击者C，如果使用老的ISN生成方式，有如下安全问题。
攻击过程：
    C冒充服务器B向服务器A发起链接请求。
    1) C 6666端口向A 8888端口发起链接请求，此时数据包内容为(源IP为B，源端口为6666,目的IP为A,目的端口为8888， ISN为C生成的ISN)
    2) C 启动9999端口向A 10000端口发起链接请求，此时数据包内容为(源IP为C，源端口为9999,目的IP为A,目的端口为10000， ISN为C生成的ISN)
    3) A收到第二个SYN包后会向C发送SYN+ACK此时数据包内容为(源IP为A，源端口为10000，目的IP为B，目的端口为9999，ACK号为C生成的ISN+1，A发送的ISN为由时间递增的序列号）
    4) A收到第一个SYN包后会向B发送SYN+ACK此时数据包内容为(源IP为A，源端口为8888，目的端口为B，目的端口为6666，ACK号为C生成的ISN+1，A发送的ISN为由时间递增的序列号）这个包C无法收到。但是C可以根据3⃣️中得到的ISN号计算出当前的ISN号.
    5) C 冒充B向A发送ACK，此时内容为(源IP为B，源端口为6666,目的IP为A，目的端口为8888，ACK为计算出的ISN号+1)
    此时链接建立。

一次完整的网站访问流程

发表于 2018-01-19 | 分类于 技术 |

当你在浏览器输入www.baidu.com，按下回车之后发生了什么？从技术的角度尽可能完整的描述技术点

查询DNS信息

1、浏览器获取本地dns缓存、读取hosts文件
2、通过UDP协议将该域名发送到local DNS请求解析的IP，local DNS查询自己本地是否有已经缓存好的www.baidu.com的域名信息，如果有则返回，如果无则local DNS代替客户端以迭代的方式进行解析(默认)。首先请求管理.com域名的DNS服务器地址，.comDNS服务器回复管理baidu.com域名的DNS服务器地址，管理baidu.com的DNS回复www.baidu.com对应的IP地址，或者回复一个cname域名，如果回复一个cname域名那么将会重新执行以便上述的流程。local DNS会根据该域名配置的ttl参数来决定缓存该解析的时间。(有一部分DNS服务会遵循自己的配置，不遵循ttl，会导致域名迁移时长尾流量，长时间不能被迁移走)。假设解析到的IP地址为1.1.1.1

访问该网站

tcp建联

客户端启动一个随机端口(例如50000端口)访问1.1.1.1:80端口
1、发送客户端发送SYN到某台机器上(可能经过四层负载均衡)，此时客户端状态为SYN_SEND状态。
2、服务端收到SYN后回复SYN+ACK，服务器进入SYN_RECV状态
3、客户端回复ACK(双方进入ESTAB状态)

http 发送get请求

tcp建联之后客户端发送get请求到服务器上，其中包含host字段”www.baidu.com”、uri、http协议版本等信息

server端响应

server端收到http请求后，根据host、uri等信息返回指定的内容。一般会保持长链接。

linux 启动流程

1:BIOS通过CMOS获取硬件各项指标
    系统通过BIOS加载CMOS，并由CMOS提供的信息取得服务器的各项硬件指标(例如硬盘信息、系统BIOS时间等)
2:BIOS自检
3:选择启动项(光盘、网络、硬盘、U盘等)
4:第一个被选择启动装置的mtr被加载到内存中执行，这段引导程序最大的作用是加载bootloader
5:bootloader的作用是：由于各个操作系统的文件系统不同，所以需要一个启动管理程序来获取操作系统的核心文件
     bootloader重要功能
        提供菜单：使用者可以选择不同的启动项目，这也是多重启动的重要功能！
        加载核心文件：直接指向可启动的程序区段来开始操作系统；
        转交其他 loader：将启动管理功能转交给其他 loader 负责。
6:bootloader将内核(vmlinuz)解压缩至内存并在此对系统各个硬件做检查，内核此时正式接替了BIOS的工作。
7:内核将initrd动解压缩到内存中，并将其模拟成根目录，加载启动过程中所需要的核心模块
8: 启动第一个程序 /sbin/init 初始化系统
9:根据/etc/inittab文件中配置选择开机启动级别
10: 执行 /etc/rc.d/rc.sysinit配置系统所需要的环境
11:根据/etc/rcX.d/下启动项将开机自启动服务启动，以及关闭在当前级别下应关闭的服务，该目录下S开头为Start启动，K开头为kill关闭。
12:调用/etc/rc.local脚本
13:弹出登录框
14:用户登录时根据/etc/passwd校验密码并获取uid,gid,bash,家目录等
15:加载/etc/bashrc /etc/bash.profile ~/.bashrc ~/.bashprofile 获取环境变量

docker 学习笔记

一、 概念篇

1.1 docker 背后的内核知识

1.1.1 资源隔离

docker使用namespace、cgroup、copy-on-write分别实现了资源隔离、资源限制、高效的文件操作。

在同一个namespace下的进程可以感知彼此的变化，而在不同的namespace下的进程无法感知。

进行NS API包括：clone()、setns()、unshare()还有/proc下的部分文件。

  • clone方法用于创建一个独立的NS进程。

  • 查看/proc/[pid]/ns文件

    作用：
    1、如果两个进程指向的ns编号相同，则说明在同一个NS下
    2、一旦上述的文件描述符被打开，就意味着该NS下的所有进程都已经结束，这个NS也会一直存在，后续进程也可以加入进来。(docker使用文件描述符来定位并加入一个存在的NS)

  • 通过setns()来加入一个已经存在的ns

    步骤：
    1、获取NS的文件描述符，用于进入到NS。
    2、使用clone()创建子进程继续执行命令，并让原进程结束运行。(setns)
    3、进入到NS，执行命令(一般会执行/bin/bash来拉起一个新的shell)

  • 通过unshare()在原进程上进行NS隔离

    主要作用：
    1、不启动新进程的情况下进行资源隔离。(在原NS上进行操作，docker中未使用该方法)

  • fork系统调用

    当程序调用fork函数时候，系统会创建新的进程，并为其分配资源。
    特性：
    仅仅被调用一次，确有两次返回。通过返回值的不同来判断哪个进程是父进程，哪个是子进程。(父进程返回子进程的PID，子进程正确创建时返回0，子进程创建失败时返回负数)

6种资源隔离

  • UTS(UNIX Time-sharing System)提供主机名、域名的隔离
  • IPC(Inter-Process Communication)进程间通信资源隔离
    主要特性：
    1、不同IPCNS下进程无法相互感知。
    2、容器与宿主机、容器与容器间。
  • PIDNS
    内核为所有的NS维护了一个树状结构，最顶层的时系统初始的时候创建的(root namespace)，后创建pid namespace被称为child namespace,父NS可以看到子NS的进程，并可以发送各种信号，子进程则不可以。
    主要特性：
    1、每一个PIDNS中都有一个PID为1的进程，该进程和传统init进程有相同的特性。
    2、每个PIDNS都不可能影响到其他PIDNS中的进程(root namespace)除外。
    3、在PIDNS中，若出现孤儿进程，该进程会交付给该PIDNS下的init进程，而并非会交付给root namespace下的init进程来管理。
    4、在PIDNS中的init进程，具有信号屏蔽功能，防止init进程被误杀。

图

图的定义

图：图是由顶点的有穷非空集合和顶点之间边的集合组成。通常表示为：G(V，E)其中，G表示一个图，V是G中顶点的集合，E是图G中，边的集合。

图就是由一堆点和一堆线组成，其中图中的元素叫做顶点，图中不可以没有元素，各个顶点之间的逻辑关系由边来表示。

图的类型

  • 简单图
    定义： 若不存在顶点到其自身的边，且不存在重复的边，被称之为简单图。

  • 无向图

    无向边： 顶点V1–>V2之间的边没有方向，则称这条边(V1,v2)为无向边。表示为(V1, V2)或者(V2, V1)。
    无向边用小括号表示。
    无向图定义： 在图中任意两个顶点之间的边都是无向边。

    • 无向完全图
      定义： 在无向图中，每个顶点之间都存在一条无向边称为无向完全图。无向完全图由n*(n-1)/2条边。

  • 有向图
    有向边： 顶点V1–>V2之间的边有方向，假设由V1指向V2，则这条边被称为有向边或者弧，V1称为弧头，V2称为弧尾。表示为”< V1, V2>”，不可写为’(V2, V1)’。

    有向边用”<>”表示。

    有向图： 在图中任意两个顶点之间的边都是有向的。

    • 有向完全图
      定义： 在任意两个顶点之间都存在两条方向互为相反的弧，则称改图为有向完全图。

网： 图的边带有权重，叫做网。

子图： 存在两个图，G1(V1, {E1})和G2(V2, {E2}),当V1包含于V2且E1包含于E2，则说明G1是G2的子图。

图的边与顶点的关系

  • 无向图
    对于无向图G(V, {E}),如果顶点V1, V2之间的边(V1, V2)包含于E， 则称两个顶点相邻接。边(V1, V2)依附于顶点V1，V2。

    度（TD）：顶点V1的度是和V1相关联边的数目。

  • 有向图
    对于有向图G(V, {E}),如果弧包含于E，则称顶点V1临接到V2，或者称顶点V2临接于V1。弧和V1,V2相关联。

    出度（OD）：以V为尾的弧的数目称之为V的出度。
    入度（ID）：以V为头的弧的数目称之为V的入度。
    度（TD）：OD+ID

简单回路（环）：从第一个顶点到最后一个顶点的路径叫做回路或者环，如果这个回路中除了第一个顶点和最后一个顶点之外没有重复的回路，被称之为简单回路。（小时候玩的一笔画完称之为简单回路（差别是不需要全部顶点都包含在内））

连通图：在图G中如果在图中任意两个顶点都是连通的，被称之为连通图。

连通分量：无向图中极大连通子图称为连通分量。

  • 要是子图
  • 子图要连通
  • 连通的子图需要包含能连通的所有顶点
  • 具有极大顶点数的连通子图包含依附于这些顶点的所有边

强连通图：有向图中每一对V1、V2，从V1–>V2、V2–>V1都存在路径，则称图G是强连通图。

强连通分量：有向图中的极大强连通子图称为强连通分量。

  • 要是子图
  • 子图要连通
  • 连通的子图需要包含能连通的所有顶点
  • 具有极大顶点数的连通子图包含依附于这些顶点的所有弧

无向连通图的生成树：就是把一个连通图用树的方式展现出来。包含全部n个顶点，并且边数为n-1

有向图的生成树：有向图中如果有一个顶点入度为0，其余入度都为1，则该图是一颗有向生成树。

有向图的生成森林：一个有向图由若干课数组成的森林，该森林包含所有顶点。

图的遍历

深度优先遍历

从某个顶点开始，依次访问与此顶点连通的顶点且未被遍历过的顶点，若下个遍历有多个选择，则优先访问右侧顶点，若所有顶点都被遍历过，则返回到未被访问到的顶点继续遍历。

数

数的定义:数是n个节点的有限集，n=0时称为空树，在任意一颗非空数中，有且仅有一个特定的称为根的结点。当n>1时，其余结点可分为m(m>0)个互不相交的有限集T1、T2….Tm,其中每个集合本身又是一棵树并且称为根的子数。

  • n>0时，根结点是唯一的，不可能存在多个根结点。
  • m>0时，子数的个数没有限制，但是它们一定是互不相交的。

结点拥有的子树数称为结点的度。度为0的结点称为叶结点或终端结点。除根结点之外，分枝结点也称为内部结点，数的度是数内各结点的最大值。

结点间关系:Child、Parent、Sibling(兄弟)

结点的层次：树中结点最大的层次称为数的深度或者高度。

如果将书的各子树看成是从左至右有次序的，不能互换的称为有序树。否则称为无序树

森林：是m(m>0)课互不相交数的集合。

线性表和数结构的不同：

线性表：

  • 第一个数据元素：无前驱
  • 最后一个数据元素：无后继
  • 中间元素：一个前驱一个后继

数：

  • 根结点：无Parent，唯一
  • 叶结点：无Child，可以多个
  • 中间结点：一个Parent多个Chile

数的存储结构

  • Parent表示法
  • Child表示法
  • Sibling表示法

Parent表示法：每个结点中，附设一个指示器指示其Parent到链表的位置。根结点的指针域设置为-1。{数据域、指针域}
数据域：存储结点数据信息
指针域：存储该结点Parent在数组中的下标

Parent表示法的好处：可以方便的根据Child结点找到Parent，但是只能以遍历的方式查找某结点的Child

Child表示法：每个结点的Child结点排列起来，以单链表作为存储结构，即n个结点有n个Child链表，如果是叶结点则次单链表为空。n个头指针又组成一个线性表，采用顺序存储结构，存放在一个一维数组中。

Sibling表示法：任意一棵树，它的结点的第一个Child如果存在就是唯一的，它的右兄弟如果存在也是唯一的，因此可以设置两个指针，分别指向该结点的第一个Child和右兄弟。

二叉树的定义

二叉树：是n（n>=0）个结点的有限集合，该集合或者为空集（空二叉树），或者有一个根结点和两颗互不相交的、分别称为根结点的左子树和右子树的二叉树组成。

二叉树的特点：

  • 每个结点最多右两颗子树，所以二叉树中不存在度大于2的结点。
  • 左子树和右子树是有顺序的，次序不能颠倒。
  • 即使某结点只有一颗子树，也要区分是左子树还是右子树。

二叉树的基本形态：

  • 空二叉树
  • 只有一个根结点
  • 根结点只有左子树
  • 根结点只有右子树
  • 根结点既有左子树又有右子树

特殊二叉树

  • 斜数：只有左结点或者只有右结点的二叉树
  • 满二叉树：在一颗二叉树中，如果所有分支结点都存在左子树和右子树，并且所有叶子都在同一层上。
    • 特点：叶子只能出现在最下一层。
    • 非叶子结点的度一定是2
    • 同样深度的二叉树中，满二叉树的结点个数最多。叶子数最多。
  • 完全二叉树
    对于一颗具有n个结点的二叉树按照层次编号，如果编号为i(1<=i<=n)的结点与同样深度的满二叉树中编号为i的结点在二叉树的位置相同，则称为完全二叉树。
    • 完全二叉树的特点
      • 叶子结点只能出现在最下两层
      • 最下层的叶子一定集中在左部连续位置
      • 倒数二层、若有叶子结点，一定都在右部连续位置。
      • 如果结点度为1，则该结点只有左子树、不存在有右子树的情况。
      • 同样结点树的二叉树，完全二叉树的深度最小。

二叉树的性质

  • 性质1: 在二叉树的第i层上最多有（2的i-1次方）个结点。
  • 性质2: 深度为k的二叉树最多有（2的k次方-1）个结点。
  • 性质3: 对于任何一个二叉树、如果叶结点的个数为(n0),度为2的结点树为(n2),则(n0)=(n2)+1
  • 性质4: 具有n个结点的完全二叉树的深度为|log2(n)|+1 #(|x|为不大于x的最大正整数。)
  • 性质5: 没看懂

二叉树的存储结构

顺序存储：用一维数组来存储结点的存储位置，不存在的结点设置为^标志，代表该结点为空。该存储结构适用于完全二叉树。极限来讲如果一个二叉树，全部为右结点，则浪费非常大的空间。如果为K个结点则需要分配（2的k次方）-1个存储单元。
二叉链表：结构体有两个指针域，分别指向左子树和右子树，还有一个data域用于存储数据。如果需要可以在增加一个指针来指向父结点。

遍历二叉树

遍历二叉树的定义：从根结点出发按照某种次序，必须对树中每个结点有且仅有访问一次。

遍历二叉树主要有四种算法：

  • 前序遍历
  • 中序遍历
  • 后序遍历
  • 层序遍历

前序遍历：每次遍历的时候从根结点只访问最左侧的结点直到最低部。当最左侧为空的时候开始由下至上访问右子树。
中序遍历：从树最左侧的底部开始访问，然后访问该结点的父结点，然后访问该结点的右结点（如果存在），如果不存在则访问父结点的父结点。当遍历到根结点时，以相同的方法遍历右子树。
后序遍历：从树的最左侧的底部开始访问，先访问该结点的右结点（如果存在），在访问该结点的父结点，然后访问该结点父结点的最左测的子树，依次类推。
层序遍历：在同一层中从左到右，从上到写依次遍历。

已知一个二叉树的前序和中序是可以确定一个二叉树。
已知一个二叉树的后序和中序也可以确定一个二叉树。
但是已知二叉树的前序和后序则不能确定一个二叉树。

线索二叉树

概念：当使用二叉链表这个存储结构时，当有n个结点的二叉链表，就会有2n个指针域，但是实际使用的只有(n-1)个分支线数，所以浪费掉了(n+1)个指针。这些空间都使用^号来表示，浪费的内存的空间。于是把这些浪费的指针空间设定为该结点的前驱指针和后继指针，这两个指针被称之为线索，拥有线索的二叉链表被称之为线索二叉树。

ltag 和 rtag:用于区分该指针为实际指针或者线索指针的标记。用布尔值来表示。

  • 当ltag为0时，指向左子树，否则指向该结点的前驱。
  • 当rtag为0时，指向右子树，否则指向该结点的后继。

布尔型变量占用内存空间小于指针变量。
注：线索的过程必须在遍历二叉树之后才能知道未使用的指针应该指向哪里。所以线索化的过程就是遍历的过程。

赫夫曼树(最优二叉树)

路径长度：从树中一个结点到另一个结点的分支构成两个结点之间的路径、路径上的分支数目叫做路径长度。

树的路径长度：就是从树根到每一个结点的路径长度之和。

赫夫曼树定义：如果考虑带权的结点、那么数的带权路径长度为各个结点的权数乘以路径长度的和，当这个数值最小时，被称之为赫夫曼树。

赫夫曼树的生成方法：根据权重进行排列组合，每次选取最小的两个结点，将他们合并为一个新的结点，其中较小的一个作为新结点的左子树，将新生成的结点代替老的两个结点继续生成一个新的结点，一直到全部生成。

赫夫曼编码：赫夫曼树主要用于数据传递，传递字符串时可通过每个字符出现的频率来生存一个赫夫曼二叉树，将编码原来的统一字符串编码改为由根结点到各个叶结点的路径(左子树标为0右子树标为1)生成一组新的编码。新生成的编码长度小于统一字符串编码发送时所需要的长度。

RHEL-Debian包管理文档

ubuntu 软件包使用

之前一直在RHEL系列工作，安装软件系列的操作十分熟悉了，比如rpm、yum命令都是使用的非常熟悉。但是debian的apt-get、dpkg确很生疏，今天有空记录一下RHEL下经常使用的操作，并对应Debian系列如何使用。

Debain

dpkg

安装软件

$ dpkg -i <package_name> #保留配置
$ dpkg -P <package_name> #不保留配置

卸载软件

$ dpkg -r <package_name>

查看包的相信信息

$ dpkg -s <package_name>

查询某文件是由哪个包生成的

$ dpkg -S <filename>

查询包里安装里哪些文件

$ dpkg -L <package_name>

查询某包是否被安装

$ dpkg -l <package_name>
$ dpkg -l | grep <package_name>

列出所有被安装的包

$ dpkg -l
apt-get命令

安装包

$ apt-get install <package_name>

指定安装包版本

$ apt-get install <package_name>=1.1.1

卸载软件

$ apt-get remove <package_name> #保留配置
$ apt-get remove --purge <package_name> #删除配置
$ apt-get autoremote #删除包以及其依赖的软件包
$ apt-get autoremote --purge #删除包以及其依赖的软件包并删除配置文件

删除本地缓存

$ apt-get clean
$ apt-get autoclean

更新和升级

$ apt-get upgrade <package_name>

查询某个文件哪个包存在

$ apt-file search <filename>

查询某个包在源中的版本

$ apt-cache show   <package_name>

RHEL下软件包管理

rpm 命令

安装软件

$ rpm -ivh <package_name>

卸载软件

$ rpm -e <package_name> ( 如果有依赖关系，yum命令会更简单一点。)

查询某文件是由哪个包生成的

$ rpm -qf <filename>

查询包里安装里哪些文件

$ rpm -qf <package_name>

查询某包是否被安装

$ rpm -q <package_name>

列出所有被安装的包

$ rpm -qa

升级一个包

$ rpm -Uvh <package_name>

$ rpm -Uvh --oldpackage <package_name> #降级
yum命令

安装包

$ yum install <package_name> #自动安装依赖包

安装包组

$ yum groupinstall <group_name>

卸载软件

$ yum remove <package_name>

删除本地缓存

$ yum clean all

查询源中有哪些包

$ yum list

更新和升级

$ yum update / yum upgrade  #全部更新

$ yum update <package_name> # 更新单个包以及所依赖的包

yum update 和 yum upgrade 的区别(google 找的并未测试)

yum upgrade == yum update --obsoletes
yum upgrade  会升级包到最新到版本，举个例子，foo 新版改名为 foo2，那么它会安装 foo2
并且把 foo 删除，而 yum update 只会升级 foo，不会将包更新到 foo2。
因此yum update 可以更保守的管理包，但是yum upgrade 会保持包的状态是最新的。

查询某个文件哪个包存在

$ yum provides "*/<filename>"
